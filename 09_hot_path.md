Самым нагруженным участком будет Betting Engine в связке с Wallet при размещении ставки (hot path). Вот почему:

## Критический участок: Размещение ставки

### Почему именно этот путь?

1\. Концентрация операций Каждая ставка вызывает каскад синхронных операций:

-   Проверка баланса (Wallet) → чтение из БД/Redis
-   Проверка статуса рынка (Market) → чтение из БД/Redis
-   Резервирование средств (Wallet) → запись с блокировкой
-   Обновление пула рынка (Betting) → запись с блокировкой
-   Пересчет odds (Betting) → CPU + запись
-   Рассылка обновлений (WebSocket/SSE) → тысячи соединений

2\. Конкуренция за ресурсы (Race Conditions) При пиковой нагрузке 300-500 RPS на запись:

-   Множество пользователей ставят на один и тот же рынок одновременно
-   Нужно атомарно обновлять пул и пересчитывать odds
-   Любая ошибка = потеря денег или неконсистентные данные

3\. Read Amplification Одна ставка → обновление odds → рассылка тысячам подключенных клиентов:

-   При 10 млн DAU одновременно онлайн могут быть ~500k-1M пользователей
-   Каждое изменение odds нужно разослать всем, кто смотрит этот рынок
-   Это создает огромную нагрузку на WebSocket-инфраструктуру

4\. Требования к консистентности

-   Деньги не должны потеряться или создаться из воздуха
-   Odds должны быть одинаковыми для всех пользователей в один момент времени
-   Нельзя принять ставку, если рынок уже закрыт или баланс недостаточен
