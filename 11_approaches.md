### 1\. Шардирование по market\_id

Разделить рынки на шарды, чтобы ставки на разные рынки не конкурировали за блокировки

шардируем не саму таблицу `market` (она маленькая, ~5000 строк, влезет в одну БД), а связанные с ней таблицы, которые накапливают миллионы записей:

-   `market_pools` (текущие пулы ставок по исходам)
-   `bets` (история ставок)
-   `odds_history` (история изменения коэффициентов)

### 

1\. Локализация транзакций Все операции по одному рынку (прием ставки, обновление пула, пересчет odds) должны выполняться атомарно в рамках одного шарда. Если шардировать по `user_id`, то ставка на один рынок может задеть десятки шардов — это ад для распределенных транзакций.

2\. Равномерное распределение (при хэшировании) Если использовать `hash(market_id) % N`, то даже популярные и непопулярные рынки равномерно размажутся по шардам.

3\. Простота запросов Когда пользователь смотрит конкретный рынок, мы точно знаем, в какой шард идти (по `market_id`). Не нужно делать scatter-gather (опрос всех шардов).

### 3\. Агрегация обновлений odds

### 

Не рассылать каждое микроскопическое изменение odds:

-   Агрегировать изменения за 100-500мс
-   Отправлять батчами через WebSocket
-   Использовать Redis Pub/Sub для масштабирования
